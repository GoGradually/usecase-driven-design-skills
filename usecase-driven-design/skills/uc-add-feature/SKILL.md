# 기존 프로젝트에 새 기능 UseCase 초안 만들기

## 개요

이미 존재하는 프로젝트에 새로운 기능의 UseCase를 추가한다.
기존 문서를 **먼저 읽고 분석**한 뒤, 8단계 워크플로우를 **파일 기반**으로 진행한다.
기존 내용과의 **일관성**과 **ID 연속성**을 보장하는 것이 핵심이다.

---

## 파일 기반 출력 규칙 (핵심)

**이 스킬은 클로드 코드 환경에서 사용한다.** 모든 산출물은 채팅이 아닌 파일에 기록한다.

### 원칙

1. **채팅에는 안내만, 내용은 파일에.** 산출물 본문을 채팅에 출력하지 않는다.
2. **단계별 개별 파일.** 각 단계의 산출물은 별도 `.md` 파일로 저장한다.
3. **파일로 확인 요청.** 단계 완료 시 "파일을 확인해주세요"로 안내한다.
4. **파일을 직접 편집.** 수정 요청이 오면 해당 파일을 편집한다.
5. **최종 통합.** 8단계 완료 후 단계별 파일을 하나의 초안으로 합친다.

### 채팅 출력 패턴

```
# 단계 완료 시 (예시)
1단계 유스케이스 목록을 작성했습니다.
→ docs/usecase/drafts/[feature]/step1-usecases.md

기존 UC 범위(UC-01~UC-05)에 이어 UC-06부터 부여했습니다.
확인 후 피드백 주세요.
```

**절대 하지 않을 것:** 표, 다이어그램, 시나리오 본문을 채팅에 직접 출력하는 것.

## 시작 절차

### 1. 프로젝트 디렉토리 확인 및 경로 판별

```bash
ls docs/usecase/
ls docs/usecase/drafts/
```

프로젝트 상태에 따라 3가지 경로로 분기한다:

```
                     docs/usecase/ 존재?
                      ┌─── Yes ───┐
                      │           │
                 경로 A (문서 분석) │
              문서 분석 → step0   │
                                 No
                      ┌──── 소스 코드 존재? ────┐
                      │                        │
                     Yes                       No
                      │                        │
                 경로 B (코드베이스 분석)   경로 C: uc-new-project
              코드 분석 → step0            안내
                      │
                 사용자 확인
                      │
                Step 1~8 (기존과 동일)
```

**경로 A**: `docs/usecase/`가 있으면 → 아래 "2-A. 기존 문서 분석" 으로 진행
**경로 B**: `docs/usecase/`가 없고 소스 코드가 있으면 → 아래 "2-B. 코드베이스 분석" 으로 진행
**경로 C**: `docs/usecase/`도 없고 소스 코드도 없으면 → `uc-new-project` 사용을 안내한다

**소스 코드 존재 판별 기준** — 다음 중 하나라도 있으면 소스 코드가 있는 것으로 판단한다:
- 빌드 매니페스트: `package.json`, `pyproject.toml`, `Cargo.toml`, `go.mod`, `build.gradle`, `pom.xml`, `*.sln`
- 소스 디렉토리: `src/`, `lib/`, `app/`, `cmd/`, `pkg/`, `internal/`

### 2-B. 코드베이스 분석 → step0 파일로 저장

`docs/usecase/`가 없고 소스 코드만 있는 프로젝트에서 기능을 추가할 때 사용한다.
기존 코드를 분석하여 엔티티, 상태 모델, API 구조를 파악한 뒤 step0을 생성한다.

#### Phase 1: 프로젝트 구조 파악 + 규모 추산

Glob으로 소스 파일을 탐색하여 언어, 프레임워크, 계층 구조를 파악하고 규모를 추산한다.

```
Glob 결과로 소스 파일 수를 집계하여 규모를 분류한다:

| 규모 | 소스 파일 수 | 분석 전략 |
|------|-------------|----------|
| Small | ~50 | 전수 분석: 모든 소스 파일을 Phase 2~5에서 직접 탐색 |
| Medium | 51~200 | 핵심 계층 분석: models/entities, routes/controllers, services 디렉토리 우선 탐색. 유틸/헬퍼/테스트는 제외 |
| Large | 201~ | 진입점 분석: 빌드 매니페스트(의존성)→ 라우터/엔트리포인트 → 참조되는 모델/서비스만 추적. step0에 "분석 범위" 섹션을 추가하여 탐색한 디렉토리와 제외한 디렉토리를 명시 |

규모 분류와 선택된 전략을 step0 "프로젝트 기술 스택" 테이블에 기재한다.
```

#### Phase 2: 엔티티 추출

Grep으로 엔티티 관련 패턴을 검색한다. Large 규모 시 Phase 1에서 식별한 핵심 디렉토리만 대상으로 한다.

검색 패턴 예시:
- `class.*Model`, `@Entity`, `@dataclass`, `type.*struct` (엔티티/모델 정의)
- `ForeignKey`, `@ManyToOne`, `@OneToMany`, `belongs_to`, `has_many` (관계)
- `Schema`, `@Table`, `CREATE TABLE` (DB 스키마)

추출 대상: 엔티티명, 주요 속성, 엔티티 간 관계

#### Phase 3: 상태 모델 추출

Grep으로 상태 관련 패턴을 검색한다.

검색 패턴 예시:
- `enum.*Status`, `enum.*State`, `PENDING|ACTIVE|COMPLETED` 등 상수 그룹
- `.status =`, `.state =`, `transition`, `setState`

추출 대상: 상태 값, 전이 이벤트, 전이 조건

#### Phase 4: 액터·시스템 경계 추출

검색 패턴 예시:
- 역할 어노테이션: `@Roles`, `@Auth`, `role`, `permission`
- API 라우트 그룹: `@Controller`, `@router`, `app.get`, `app.post`
- 외부 시스템 클라이언트: `HttpClient`, `fetch(`, `axios`, `requests.`

추출 대상: 액터(역할), API 엔드포인트, 외부 시스템 연동

#### Phase 5: 도메인 규칙 추출

Grep으로 규칙 관련 패턴을 검색한다.

검색 패턴 예시:
- `validate`, `assert`, `ensure`, `check`, `require`
- 도메인 예외 클래스: `Exception`, `Error`, `throw`, `raise`
- 비즈니스 조건문: 주요 if/guard 절

추출 대상: 유효성 검증 규칙, 비즈니스 불변식, 도메인 예외

#### Phase 6: step0 파일 합성

분석 결과를 step0 파일로 합성한다. 경로 A의 step0와 동일한 구조에 코드 분석 고유 섹션을 추가한다.

**파일:** `docs/usecase/drafts/[새-feature]/step0-existing-analysis.md`

## 진행 규칙

1. **기존 문서와 동일한 8단계를 따른다** (uc-new-project와 동일).
2. **ID는 기존 범위 다음부터 부여한다.** (step0에서 파악한 시작점 사용)
3. **기존 액터를 재사용한다.** 새 액터가 필요한 경우에만 추가한다.
4. **도메인 모델은 기존 모델을 확장한다.** 기존 엔티티와 충돌하지 않도록 한다.
5. **상태 모델은 기존 모델을 확장한다.** 기존 상태/전이를 유지하면서 신규 상태/전이를 추가한다.
6. **각 단계를 파일에 기록하고 확인을 받는다.**

---

## Mermaid 다이어그램 규칙

uc-new-project와 동일한 Mermaid 규칙을 따른다.

**기존 프로젝트 확장 시 추가 규칙:**
- 기존 문서의 노드 ID를 재사용한다.
- 신규 요소는 기존 다이어그램에 추가하는 형태로 작성한다.
- 기존 요소와의 연결 관계를 반드시 표현한다.

### 4단계: 도메인 모델 설계 (확장)

**파일:** `docs/usecase/drafts/[feature]/step4-domain-model.md`

기존 도메인 모델을 기반으로 확장한다.

```markdown
# Step 4: 도메인 모델 (확장)

> ...

## 기존 엔티티 (변경 없음)
| 엔티티 | 비고 |
|--------|------|
| Order  | 기존 유지 |

## 기존 엔티티 (속성 추가)
| 엔티티 | 추가 속성 | 사유 |
|--------|----------|------|
| Order  | refundStatus | 환불 기능 추가로 필요 |

## 신규 엔티티
| 엔티티 | 설명 | 주요 속성 |
|--------|------|----------|
| Refund | 환불 | refundId, reason, amount |

## 신규 관계

| 엔티티 A | 엔티티 B | 다중성 | 관계 유형 | 설명 |
|---------|---------|--------|----------|------|
| Order | Refund | 1:1 | 합성(composition) | 주문 삭제 시 환불도 삭제 |

**관계 유형 정의:**
- **연관(association)**: 독립적 생명주기. A 삭제해도 B 유지
- **집약(aggregation)**: 전체-부분 관계이나 부분이 독립 존재 가능
- **합성(composition)**: 전체-부분 관계이며 부분이 전체에 종속

## 신규 일반화/특수화 (IS-A 관계)

신규 엔티티 간 또는 기존 엔티티와의 상속 관계가 있는 경우 식별한다.

| 상위 엔티티 | 하위 엔티티 | 구분 기준 | 비고 |
|------------|-----------|----------|------|
| (해당 없음) | | | |

## 신규 도메인 규칙
- [규칙]: ...

## 클래스 다이어그램
(기존 엔티티를 포함한 통합 다이어그램. Mermaid에서 연관 `-->`, 집약 `o--`, 합성 `*--`, 상속 `<|--` 사용)

## BCE 분석 (Boundary-Control-Entity)

신규 UC의 시나리오를 분석하여 세 가지 분석 객체로 분해한다.
기존 UC의 Boundary/Control은 참조만 하고, 신규 UC에 대해서만 작성한다.

### UC-[N+1]: [유스케이스명]

| 구분 | 분석 객체 | 설명 | 시나리오 근거 |
|------|----------|------|-------------|
| Boundary | 환불 요청 화면 | 사용자가 환불을 요청하는 인터페이스 | 기본흐름 1단계 |
| Control | 환불 처리기 | 환불 요청 로직을 조정 | 기본흐름 2~3단계 |
| Entity | Refund | 환불 도메인 객체 | 엔티티 목록 참조 |
| Entity | Order (기존) | 주문 도메인 객체 | 기존 엔티티 참조 |

### BCE 상호작용 다이어그램

(Mermaid sequenceDiagram — Actor → Boundary → Control → Entity 흐름)
```

**주의사항:**
- 신규 관계의 유형(연관/집약/합성)을 생명주기 의존성 기준으로 분류한다. 기존 관계 유형과의 정합성을 확인한다.
- 신규 엔티티가 기존 엔티티와 IS-A 관계를 형성하는지 검토한다.
- BCE 분석에서 기존 UC의 Boundary/Control과 신규 UC의 Boundary/Control이 중복되는지 확인한다.

**→ 파일 저장 후 확인 요청.**

### 5단계: 상태 모델 (확장)

**파일:** `docs/usecase/drafts/[feature]/step5-state-model.md`

2단계 시나리오와 4단계 도메인 모델을 읽고, 기존 상태 모델을 참조하여 **이벤트 → 전이 → 상태 도출** 순서로 확장한다.

```markdown
# Step 5: 상태 모델 (확장)

> ...

## 기존 상태 모델 참조
- Order 엔티티: 기존 상태 [CREATED, CONFIRMED, PAID, SHIPPED]
- 기존 이벤트: [주문 생성, 주문 확인, 결제 완료, 배송 시작]

## 1. 신규 비즈니스 이벤트 식별

이 기능의 시나리오에서 엔티티의 상태를 변화시키는 행위(동사)를 추출한다.

| 이벤트 | 설명 | 대상 엔티티 | UC 시나리오 근거 |
|--------|------|-----------|----------------|
| 환불 요청 | 사용자가 환불을 요청함 | Order, Refund | UC-06 기본흐름 2단계 |
| 환불 승인 | 관리자가 환불을 승인함 | Order, Refund | UC-07 기본흐름 3단계 |

## 2. 이벤트-전이 매핑 (기존 엔티티 확장)

### Order — 신규 전이

| 이벤트 | 출발 상태 | 도착 상태 | 비고 |
|--------|----------|----------|------|
| 환불 요청 | PAID (기존) | REFUND_REQUESTED (신규) | 기존 상태에서 분기 |
| 환불 승인 | REFUND_REQUESTED (신규) | REFUNDED (신규) | |

## 3. 신규 엔티티 상태 모델

### Refund — 이벤트-전이 매핑

| 이벤트 | 출발 상태 | 도착 상태 |
|--------|----------|----------|
| 환불 요청 | [초기] | REQUESTED |
| 환불 승인 | REQUESTED | APPROVED |

### Refund — 상태 확정

| 상태 | 설명 | 진입 이벤트 |
|------|------|-----------|
| REQUESTED | 환불이 요청됨 | 환불 요청 |
| APPROVED | 환불이 승인됨 | 환불 승인 |

## 4. 불허 전이

| 엔티티 | 출발 상태 | 도착 상태 | 불허 사유 |
|--------|----------|----------|----------|
| Order | REFUNDED | PAID | 환불 완료된 주문은 결제 상태로 되돌릴 수 없음 |

## 5. 상태 다이어그램
(Mermaid stateDiagram-v2 — 기존 상태는 연한 스타일, 신규 상태는 강조 스타일)

## 6. 기존 UC 영향 분석

신규 상태/전이 추가로 인해 기존 UC의 흐름이 변경되는 경우를 정리한다.

| 기존 UC | 영향받는 전이 | 변경 내용 |
|---------|-------------|----------|
| UC-02 (주문 확인) | CREATED→CONFIRMED | 사전조건에 "환불 요청 상태가 아닐 것" 추가 필요 |
```

**주의사항:**
- **이벤트가 출발점이다.** 신규 시나리오의 동사에서 이벤트를 식별하고, 이벤트가 일으키는 전이를 통해 신규 상태를 도출한다.
- 기존 상태 모델의 상태와 이벤트를 먼저 참조하고, 신규 이벤트/전이/상태만 추가한다.
- 신규 이벤트가 기존 상태를 출발점으로 삼을 수 있다.
- Mermaid 다이어그램에서 기존 상태는 연한 스타일로, 신규 상태는 강조 스타일로 구분한다.
- **기존 전이 그래프에 신규 상태를 삽입하는 위치를 명시한다.**
  step0에서 파악한 기존 전이 전체도를 기준으로, 신규 이벤트가 기존 상태 흐름의
  정확히 어느 지점(어느 상태 이후, 어느 전이 대신/분기)에 개입하는지를 서술한다.
- **기존 UC의 시나리오에 미치는 영향을 기록한다.**
  신규 상태가 삽입되면 기존 UC의 사전조건/사후조건이 바뀔 수 있다.
  영향받는 기존 UC ID를 목록으로 남긴다.

**→ 파일 저장 후 확인 요청.**
