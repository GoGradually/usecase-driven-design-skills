# 유스케이스 주도 설계 워크플로우

## 개요

유스케이스 주도 설계를 **7단계**로 나누어 진행한다.
각 단계의 산출물을 **개별 마크다운 파일**에 기록하고, 사용자가 파일을 확인한 뒤 다음 단계로 넘어간다.
완료 후 **하나의 통합 설계서**로 합친다.

---

## 파일 기반 출력 규칙 (핵심)

**이 스킬은 클로드 코드 환경에서 사용한다.** 모든 산출물은 채팅이 아닌 파일에 기록한다.

### 원칙

1. **채팅에는 안내만, 내용은 파일에.** 산출물 본문을 채팅에 출력하지 않는다.
2. **단계별 개별 파일.** 각 단계의 산출물은 별도 `.md` 파일로 저장한다.
3. **파일로 확인 요청.** 단계 완료 시 "파일을 확인해주세요"로 안내한다.
4. **파일을 직접 편집.** 수정 요청이 오면 해당 파일을 편집한다.
5. **최종 통합.** 7단계 완료 후 단계별 파일을 하나의 설계서로 합친다.

### 채팅 출력 패턴

```
# 단계 완료 시 (예시)
1단계 유스케이스 목록을 작성했습니다.
→ docs/usecase/step1-usecases.md

확인 후 피드백 주세요. 수정할 부분이 있으면 말씀해주시고,
괜찮으면 다음 단계로 진행하겠습니다.
```

**절대 하지 않을 것:** 표, 다이어그램, 시나리오 본문을 채팅에 직접 출력하는 것.

## 단계별 파일 공통 헤더

```markdown
# Step N: [단계명]

> 프로젝트: [project-name]
> 작성일: [날짜]
> 상태: ✏️ 작성중 | ✅ 확인완료
```

---

## 진행 규칙

1. **반드시 순서대로 진행한다.** 단계를 건너뛰지 않는다.
2. **각 단계를 파일로 저장한 뒤 확인을 요청한다.**
3. **사용자가 수정을 요청하면** 해당 파일을 직접 편집하고 다시 확인을 요청한다.
4. **이전 단계의 파일을 다음 단계 작성 시 참조한다.**
5. **7단계 완료 후 통합 파일을 생성한다.**

---

## 7단계 워크플로우

### 1단계: 유스케이스 작성

**파일:** `docs/usecase/step1-usecases.md`

사용자에게 구현할 기능/시스템 정보를 수집하고 유스케이스를 식별한다.

**파일 내용:**

```markdown
# Step 1: 유스케이스 목록

> 프로젝트: [project-name]
> 작성일: [날짜]
> 상태: ✏️ 작성중

## 유스케이스 목록

| ID | 유스케이스명 | 주요 액터 | 목적 (한 줄 설명) |
|----|-------------|----------|------------------|
| UC-01 | ... | ... | ... |
| UC-02 | ... | ... | ... |
```

**참고:** 전체 버전(uc-new-project)에서는 시스템 경계 단계에서 Mermaid로 유스케이스 다이어그램을 작성한다.

**→ 파일 저장 후 확인 요청. 확인되면 상태를 ✅로 변경하고 2단계로 진행.**

---

### 2단계: 유스케이스 시나리오 작성

**파일:** `docs/usecase/step2-scenarios.md`

1단계 파일을 읽고, 각 UC의 기본 흐름과 대안 흐름을 작성한다.

**파일 내용:**

```markdown
# Step 2: 유스케이스 시나리오

> ...

## UC-01: [유스케이스명]

### 기본 흐름 (Main Flow)
1. 액터가 ...한다.
2. 시스템이 ...한다.
3. ...

### 대안 흐름 (Alternative Flow)
- **2a.** [조건] → 시스템이 ...한다.

---

## UC-02: [유스케이스명]
(동일 형식)
```

**주의사항:**
- "누가 무엇을 한다" 형식으로 주어를 명확히 한다.
- 기본 흐름은 성공 경로, 대안 흐름은 분기를 기술한다.

**→ 파일 저장 후 확인 요청.**

---

### 3단계: 도메인 모델 설계

**파일:** `docs/usecase/step3-domain-model.md`

시나리오의 핵심 개념(명사)을 추출하여 도메인 모델을 설계한다.

**파일 내용:**

```markdown
# Step 3: 도메인 모델

> ...

## 엔티티 목록

| 엔티티 | 설명 | 주요 속성 |
|--------|------|----------|
| Order  | 주문 | orderId, status, createdAt |

## 관계

- Order → OrderItem: 1:N (하나의 주문은 여러 주문항목을 가진다)

## 도메인 규칙

- [규칙 1]: ...
```

**→ 파일 저장 후 확인 요청.**

---

### 4단계: 상태 모델

**파일:** `docs/usecase/step4-state-model.md`

2단계 시나리오와 3단계 도메인 모델을 읽고, **이벤트 → 전이 → 상태 도출** 순서로 개념 수준의 상태 모델을 확정한다.

**파일 내용:**

```markdown
# Step 4: 상태 모델

> 프로젝트: [project-name]
> 작성일: [날짜]
> 상태: ✏️ 작성중

## 1. 비즈니스 이벤트 식별

시나리오의 각 단계에서 엔티티의 상태를 변화시키는 행위(동사)를 추출한다.

| 이벤트 | 설명 | 대상 엔티티 | UC 시나리오 근거 |
|--------|------|-----------|----------------|
| 주문 생성 | 사용자가 새 주문을 생성함 | Order | UC-01 기본흐름 3단계 |
| 주문 확인 | 관리자가 주문을 확인함 | Order | UC-02 기본흐름 2단계 |

## 2. 이벤트-전이 매핑

| 이벤트 | 출발 상태 | 도착 상태 |
|--------|----------|----------|
| 주문 생성 | [초기] | CREATED |
| 주문 확인 | CREATED | CONFIRMED |

## 3. 엔티티별 상태 확정

### Order 상태 목록

| 상태 | 설명 | 진입 이벤트 |
|------|------|-----------|
| CREATED | 주문이 생성됨 | 주문 생성 |
| CONFIRMED | 주문이 확인됨 | 주문 확인 |

### Order 전이 전체도

| 출발 상태 | 이벤트 | 도착 상태 | UC 시나리오 근거 |
|----------|--------|----------|----------------|
| [초기] | 주문 생성 | CREATED | UC-01 기본흐름 3단계 |
| CREATED | 주문 확인 | CONFIRMED | UC-02 기본흐름 2단계 |

## 4. 불허 전이

| 출발 상태 | 도착 상태 | 불허 사유 |
|----------|----------|----------|
| SHIPPED | CREATED | 배송된 주문은 생성 상태로 되돌릴 수 없음 |
```

**주의사항:**
- **이벤트가 출발점이다.** 시나리오의 동사에서 이벤트를 식별하고, 이벤트가 일으키는 전이를 통해 상태를 도출한다.
- 이 모델은 **개념 수준**(비즈니스 이벤트, 비즈니스 상태)이다. 구현 수준(DB 컬럼, 이벤트 큐)은 다루지 않는다.
- 불허 전이는 비즈니스 규칙으로 명시적으로 금지되는 전이를 기록한다.

**→ 파일 저장 후 확인 요청.**

### 5단계: 시스템 경계 분리

**파일:** `docs/usecase/step5-system-boundary.md`

시스템 경계를 명확히 한다.

**파일 내용:**

```markdown
# Step 5: 시스템 경계 분리

> ...

## 외부 액터

| 액터 | 유형 | 상호작용 방식 |
|------|------|-------------|
| 사용자 | 사람 | Web UI |

## 외부 유스케이스

| ID | 유스케이스명 | 호출 액터 |
|----|-------------|----------|
| UC-01 | ... | 사용자 |

## 내부 유스케이스

| ID | 유스케이스명 | 트리거 |
|----|-------------|--------|
| UC-INT-01 | ... | UC-01의 3단계에서 호출 |
```

**참고:** 전체 버전(uc-new-project)에서는 이 단계에서 시스템 경계 다이어그램과 UML 스타일 유스케이스 다이어그램을 Mermaid로 작성한다.

**→ 파일 저장 후 확인 요청.**

### 6단계: 예외 정리

**파일:** `docs/usecase/step6-exceptions.md`

```markdown
# Step 6: 예외 정리

> ...

## UC-01: [유스케이스명]

| 시나리오 단계 | 시점 | 예외 상황 | 시스템 대응 |
|-------------|------|----------|-----------|
| 2. 시스템이 검증한다 | 전 | 입력 데이터 누락 | 400 에러 |
```

**주의사항:**
- 4단계 상태 모델의 불허 전이도 예외 상황의 근거가 된다.

**→ 파일 저장 후 확인 요청.**

---

### 7단계: 사전조건/사후조건 작성

**파일:** `docs/usecase/step7-conditions.md`

```markdown
# Step 7: 사전조건 / 사후조건

> ...

## UC-01: [유스케이스명]

**사전조건 (Precondition):**
- 사용자가 로그인되어 있어야 한다.

**사후조건 (Postcondition) - 성공 시:**
- 주문이 생성되고 상태가 "대기중"이다.

**사후조건 (Postcondition) - 실패 시:**
- 시스템 상태가 변경되지 않는다 (롤백).
```

**주의사항:**
- 사전조건에 4단계 상태 모델의 "출발 상태"를 반영한다.
- 사후조건에 4단계 상태 모델의 "도착 상태"를 반영한다.

**→ 파일 저장 후 확인 요청.**

## 최종 통합

7단계 파일이 모두 `✅ 확인완료`가 되면 통합 설계서를 생성한다.

**통합 방법:**
1. `step1` ~ `step7` 파일을 순서대로 읽는다.
2. 각 파일의 헤더(Step N, 상태 등)를 제거하고 본문만 추출한다.
3. 설계서 구조에 맞게 재구성한다.

**파일:** `docs/usecase/[project-name]-usecase-design.md`

```markdown
# [프로젝트/기능명] 유스케이스 설계서

> 작성일: [날짜]

## 1. 유스케이스 목록
(step1 내용)

## 2. 유스케이스 시나리오
(step2 내용)

## 3. 도메인 모델
(step3 내용)

## 4. 상태 모델
(step4 내용)

## 5. 시스템 경계 분리
(step5 내용)

## 6. 예외 정리
(step6 내용)

## 7. 사전조건 / 사후조건
(step7 내용)
```

통합 완료 후 사용자에게 파일 경로를 안내한다.
